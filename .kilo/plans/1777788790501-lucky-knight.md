# Plan: Export CCTNS Data to CSV Files

## Objective
Export the following data from CCTNS APIs to CSV files:
1. Complaints for April 2026 (01/04/2026 - 30/04/2026)
2. Master data: Districts, Police Stations, Offices

## Analysis of Existing Codebase

### CCTNS API Integration
The backend has mature CCTNS integration in `backend/src/services/cctns.ts`:
- `getCctnsToken()` — fetches bearer token from `ReqToken` API
- `fetchCctnsComplaints(timeFrom, timeTo)` — fetches complaint data, handles AES decryption automatically
- Token caching with 55-minute expiry
- Handles both JSON and encrypted responses

### Complaint Data Normalization
`backend/src/services/cctns-normalize.ts` provides:
- `CctnsComplaintRow` interface matching CCTNS field names
- `normalizeComplaintRow()` — converts raw rows to normalized format
- Date parsing for DD/MM/YYYY format
- Field mapping: COMPL_REG_NUM, DISTRICT, COMPL_DESC, FIRST_NAME, LAST_NAME, MOBILE, etc.

### Master Data Sync
`backend/src/services/government.ts` contains:
- `fetchGovernmentItems(url)` — fetches and parses JSON or XML from government APIs
- District API: `https://api.haryanapolice.gov.in/eSaralServices/api/common/district`
- Police Station API: `GetPSByDistrict?state=13&district=<districtId>`
- Office API: `https://api.haryanapolice.gov.in/eSaralServices/api/common/GetAllOffices`
- Handles both JSON and XML response formats

### Existing Scripts
- `backend/backfill-april-2026.ts` — fetches April 2026 complaints and saves to PostgreSQL `CCTNSComplaint` table
- `backend/sync-3-months.ts` — historical sync script
- Multiple test scripts demonstrate API connectivity

## Implementation Plan

### Step 1 — Create standalone TypeScript export script
Create `backend/export-cctns-data.ts` that:
- Imports existing service logic (`getCctnsToken`, `fetchCctnsComplaints`, `normalizeComplaintRow`)
- Reuses `fetchGovernmentItems` logic for master data
- Uses `csv-writer` or manual CSV generation with `fs`
- Takes no arguments, hardcodes April 2026 date range
- Exports 4 CSV files to `backend/exports/` directory:
  - `complaints-april-2026.csv` — all complaint fields from normalized data
  - `districts.csv` — district ID, name
  - `police-stations.csv` — ps_id, district_id, district_name, ps_name
  - `offices.csv` — office_id, office_name

### Step 2 — CSV schema design
**complaints-april-2026.csv columns (from Prisma Complaint model + normalized fields):**
- complRegNum, complRegDt, districtMasterId, policeStationMasterId, officeMasterId
- districtName, complDesc, complSrno, firstName, lastName, mobile, gender, age
- addressLine1, addressLine2, addressLine3, village, tehsil, addressDistrict, addressPs
- receptionMode, incidentType, incidentPlc, incidentFromDt, incidentToDt
- submitPsCd, submitOfficeCd, email, statusRaw, statusGroup, statusOfComplaint, disposalDate
- classOfIncident, complaintSource, typeOfComplaint, crimeCategory, complainantType, complaintPurpose
- ioDetails, respondentCategories, transferDistrictCd, transferOfficeCd, transferPsCd
- firNumber, actionTaken, branch

**districts.csv:**
- id, name

**police-stations.csv:**
- id, districtId, districtName, name

**offices.csv:**
- id, name

### Step 3 — Error handling and logging
- Wrap each API call in try-catch
- Log progress: token fetch, complaint fetch count, master data counts
- On CCTNS API failure (e.g., 500 IP block), log error and exit gracefully with message
- Master data APIs are public and should succeed reliably

### Step 4 — Dependencies
- Use built-in `fs/promises` for file writes
- No new packages required; use existing `csv-stringify` or simple `Array.map().join(',')`
- Script runs with `tsx` (already in devDependencies)

### Step 5 — Execution
From project root, run:
```bash
cd backend
npx tsx export-cctns-data.ts
```

CSV files appear in `backend/exports/` directory.

## Key Decisions

**Complaint data range:** April 2026 = 01/04/2026 to 30/04/2026 (entire month)

**CSV format:** UTF-8 encoded, comma-separated, header row included, fields wrapped in double quotes if they contain commas

**Decryption:** Use existing `decrypt()` function from `cctns.ts` if API returns non-JSON base64 response

**Master data sync:** Fetch all districts first, then fetch police stations for each district sequentially to avoid rate limits, then fetch all offices

**No database write:** Direct export to CSV; do not use Prisma or database at all

## Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| CCTNS Complaint API blocks non-whitelisted IP (returns 500) | Cannot fetch complaint data | Script catches error, logs message to contact IT cell, still exports master data CSV |
| Response format changes (XML token, encrypted JSON) | Token extraction fails | Existing CCTNS service handles multiple formats — reuse it |
| Large dataset causes memory issues | Process hang | Stream to CSV line-by-line or process in chunks of 1000 records |
| Government API rate limits | Partial master data | Fetch districts first (fast), then PS with 1-second delays between district calls |
