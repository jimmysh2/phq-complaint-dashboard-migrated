# CCTNS Route Technical Audit & Refactor Plan

## Audit Summary

### 1. Data Fetching Pipeline

**Current Flow:**
```
CCTNS API → fetchCctnsComplaints() → Frontend (Live Tab)
                     ↓
              User clicks "Sync to DB"
                     ↓
              POST /cctns/sync-complaints → collectComplaintsByRange() → normalize → upsert to DB
```

**Findings:**
- **"Fetch Live Data"** (`GET /cctns/complaints-live`) fetches from CCTNS API and returns raw data to the frontend. It does **NOT** write to the database.
- **"Sync to DB"** (`POST /cctns/sync-complaints`) is a separate, explicit step that fetches the same data again and writes to DB.
- There is **no direct-to-DB persistence** on "Fetch Live Data". The data is purely client-side display.
- The `fetchCctnsComplaints()` service has a 5-minute timeout (`AbortSignal.timeout(300000)`) and token caching.

**Flaw:** The two-step process (Fetch → then Sync) is unnecessary and violates the requirement for direct-to-DB persistence. Users must click two buttons to achieve what should be one action.

---

### 2. Asynchronous Process Management

**Current Implementation:**
- Both `/cctns/complaints-live` and `/cctns/sync-complaints` are **standard synchronous HTTP requests** handled by Fastify route handlers.
- There is **no background job queue**, **no worker process**, and **no async task management** for the sync endpoint.
- The `collectComplaintsByRange()` function iterates date ranges in 3-day windows and makes sequential API calls. For a 30-day range, this means ~10 sequential HTTP requests to the CCTNS API, each potentially slow.

**Flaw:** If the user navigates away or closes the browser during a sync:
- The HTTP request is dropped.
- The server **may or may not** continue processing depending on Node.js/Fastify behavior, but there is no guarantee of completion.
- There is **no status tracking** for the manual sync operation (unlike the background job which uses `SyncRun`).
- The `isSyncing` flag in `cctns-sync-job.ts` only protects the background job, not the manual sync endpoint.

**Background Job (Existing):**
- `cctns-sync-job.ts` runs a daily background sync via `setInterval`.
- It creates a `SyncRun` record to track status.
- This is the only async process management in place.

---

### 3. Redundancy Analysis

**Current Buttons:**
1. **"Fetch Live Data"** — Fetches from CCTNS API, displays in "Live CCTNS Data" tab. No DB write.
2. **"Sync to DB"** — Fetches from CCTNS API (same data), normalizes, upserts to DB.

**Problem:**
- The "Sync to DB" button is redundant if "Fetch Live Data" were implemented correctly.
- Both buttons trigger the same underlying API fetch. The only difference is whether the result is persisted.
- The "Live CCTNS Data" tab shows data that is not in the DB, creating a disconnect between what's displayed and what's stored.

**Proposed Resolution:**
- Remove the two-button model.
- "Fetch Live Data" should fetch **and** persist directly to DB in a single operation.
- The "Live CCTNS Data" tab should then display the most recently synced records from the DB (or show the sync-in-progress status).

---

### 4. Database Gateway Implementation ("Synced Records" Tab)

**Current Implementation:**
- `GET /cctns` returns the last 500 records: `prisma.complaint.findMany({ orderBy: { id: 'desc' }, take: 500 })`.
- Frontend shows a basic `DataTable` with 5 columns: Reg. No., Reg. Date, District, PS, Status, Disposal Date.
- No pagination, no search, no filtering, no sorting server-side.
- No visibility into sync history, errors, or audit trail.

**Flaws:**
- **Not a comprehensive gateway**: 500-record hard limit prevents full visibility.
- **No audit trail**: No way to see when records were last synced, how many were fetched, or if errors occurred.
- **No sync run history**: The `SyncRun` table exists but is not exposed via any API or UI.
- **Limited columns**: Many important fields from the `Complaint` model are hidden.
- **No CRUD operations**: The backend has `POST /cctns`, `PUT /cctns/:id`, `DELETE /cctns/:id` but the frontend doesn't use them.

---

### 5. State Synchronization ("Live CCTNS Data" Tab)

**Current Implementation:**
- Uses React Query `useQuery` with `enabled: fetchTrigger > 0`.
- **No polling** (`refetchInterval` is not set).
- **No WebSocket** or Server-Sent Events.
- Data is fetched once when the user clicks "Fetch Live Data" and then remains static.
- If the user switches to the "Synced Records" tab and back to "Live", the live data is stale.

**Flaws:**
- **Not real-time or near-real-time**: Data does not auto-refresh.
- **Stale data on tab switch**: The `liveQuery` is not invalidated when switching tabs.
- **No indication of data freshness**: No timestamp showing when the data was last fetched.
- **Background sync is invisible**: The daily background sync updates the DB but the "Live" tab doesn't reflect this.

## Proposed Refactor Plan

### Phase 1: Unify Fetch + Sync into Direct-to-DB Pipeline

**Backend Changes:**
1. **Modify `GET /cctns/complaints-live`** to also persist fetched data to DB before returning.
   - Or better: replace it with a new `POST /cctns/fetch-and-sync` endpoint that:
     - Accepts `timeFrom`, `timeTo`
     - Creates a `SyncRun` record with status `running`
     - Fetches data from CCTNS API in chunks
     - Normalizes and upserts to DB
     - Updates `SyncRun` with final status
     - Returns the sync run ID and summary
2. **Remove `POST /cctns/sync-complaints`** or repurpose it as the unified endpoint.

**Frontend Changes:**
1. Replace the two buttons ("Fetch Live Data" + "Sync to DB") with a single **"Fetch & Sync"** button.
2. Show a progress indicator while sync is running.
3. Poll the sync status endpoint to show real-time progress.

### Phase 2: Implement Background Job + Status Tracking

**Backend Changes:**
1. Create a `POST /cctns/fetch-async` endpoint that:
   - Queues the fetch job (even if just in-memory for now)
   - Returns a job ID immediately
2. Create a `GET /cctns/sync-status/:id` endpoint to poll job status.
3. Enhance the existing `runCctnsSync` to accept custom date ranges and be reusable for both background and manual triggers.

**Frontend Changes:**
1. After triggering fetch, poll for status using the job ID.
2. Show progress: "Fetching chunk 3 of 10...", "Upserting 150 records..."
3. Auto-refresh the "Synced Records" table when sync completes.

### Phase 3: Transform "Synced Records" into a Full DB Gateway

**Backend Changes:**
1. Enhance `GET /cctns` with:
   - Pagination (`page`, `limit`)
   - Search (`q` param across multiple fields)
   - Filtering (`district`, `statusGroup`, `dateFrom`, `dateTo`)
   - Sorting (`sortBy`, `sortOrder`)
2. Create `GET /cctns/sync-runs` to expose `SyncRun` history.
3. Create `GET /cctns/sync-runs/:id/details` to show per-run statistics.

**Frontend Changes:**
1. Add pagination controls to the Synced Records table.
2. Add search and filter UI.
3. Add a "Sync History" sub-section or modal showing past sync runs with status, counts, and timestamps.
4. Add a "View Details" action for each record (linking to the existing `GET /cctns/:id` endpoint).

### Phase 4: Real-Time Updates for "Live" Tab

**Frontend Changes:**
1. Since the unified fetch writes to DB, the "Live CCTNS Data" tab should show the **most recently synced records from DB**.
2. Use `refetchInterval` (e.g., 30 seconds) on the synced records query when the "Live" tab is active.
3. Add a "Last synced: 2 minutes ago" timestamp.
4. Show a banner if the last sync was more than X hours ago (stale data warning).

### Phase 5: Remove Redundancies

1. **Remove the separate "Sync to DB" button and mutation** from the frontend.
2. **Deprecate or remove `POST /cctns/sync-complaints`** from the backend.
3. **Remove the `fetchTrigger` state** and the `liveQuery` that fetches without persisting.
4. Consolidate the two tabs conceptually: "Live" shows recent DB records with auto-refresh; "Synced Records" shows the full gateway with search/filter.

## Open Questions

1. Should the unified fetch be synchronous (wait for completion) or asynchronous (return job ID immediately)?
2. For the DB gateway, what is the expected scale — how many CCTNS records are anticipated? This affects pagination strategy.
3. Should the "Live" tab be renamed to "Recent Syncs" or "Latest Records" since it will show DB data?
4. Is there a need for record-level editing/deletion in the gateway, or is it read-only?
